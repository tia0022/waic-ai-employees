## 〇. 步骤经济性原则（Step Economy）

> "每个小明尽量要控制好步骤的总数量，保证成本相对可控和效果的平衡。" —— 老板原话（2026-05-21）

| 约束 | 目标 |
|---|---|
| 单次任务工作流主步骤数 | **≤ 3 步**（每步可含 2~3 个原子动作） |
| 每个主步骤覆盖的 Skill 数 | 2~4 个（高内聚、低耦合） |
| HITL 节点 | 每个主步骤至多 1 个，全流程 ≤ 3 个 |
| 每一步的价值锚点 | 必须能回答"做了什么"+"省了什么"+"留了什么证据" |

**为什么要克制？**

1. **成本约束**：每多一步意味着多一次 LLM/Skill 调用，跨境客户日均轨迹查询/异常处置任务可达万次量级，3 步 vs 6 步 = 50% 成本差
2. **业务真实性**：供应链负责人遇到危机时，能快速看完的决策卡不超过 3 张，超过即决策失焦
3. **效果阈值**：经验上前 3 个动作能拿到 80% 处置效果，第 4~6 个动作只贡献 20%（边际递减）
4. **应急时效**：物流危机处置闭环目标 ≤ 30 分钟，6 步串联会拖慢响应

> 注：完整 Skill 清单（§2）保持 10 个不变——Skill 是"能力原子"，工作流是"调度编排"。**减步骤不是减能力，而是把能力打包到更少的调度节点里**。

## 1. 岗位职责（Job Definition）

| 字段 | 内容 |
|---|---|
| 岗位名称 | 物流小明（Logistics Xiao Ming） |
| 职级定位 | 跨境供应链调度专家 · 相当于供应链总监 + 关务专员 + 仓配协调员 + 索赔专员的复合角色 |
| 核心使命 | 把跨境物流从"事后追踪+碎片化轨迹查询"升级为"事前预测断货 + 事中实时调度 + 事后证据闭环"，交付以分钟计算的供应链处置方案，而非提供一套 TMS 软件 |
| 服务对象 | 跨境卖家供应链负责人、仓配运营专员、财务（应付账款/物流费用）；服务商旗下物流团队；保险公司理赔接口（协作方） |

**核心职能（三条）**

1. **全球头程到海外仓/FBA 全链路实时监控与 ETA 偏差预警**：对接多承运商/货代/海外仓/FBA 系统，把碎片化轨迹聚合为统一的"在途态势图"，并基于历史偏差和实时事件对每一批货预测 ETA 与异常概率。
2. **物流异常根因定位 + 多方案处置 + 跨 Agent 协同决策**：把"货延了/被查/破损/断货"等异常信号转化为"继续海运/部分空运/调拨/补发"四类候选方案，并自动联动内部分析、营销、合规、经营评估四位小明形成统一处置闭环。
3. **清关合规联动 + 索赔证据组织 + 供应链金融数据准备**：清关异常时调用合规小明的单证规则；索赔/查验场景下自动组织截图、轨迹、单证、保单证据包；供应链金融场景下输出贸易真实性证据给经营评估小明。

**不做什么（边界）**

- **不替代企业与承运商签订运输合同**——合同主体仍是企业法人
- **不直接对海关/外汇局做申诉沟通**——仅准备材料和方案，沟通由人完成
- **不做长期供应链战略规划**（选承运商组合、布仓策略）——限定在执行/调度层
- **不做仓库实物作业**（拣货/打包/上架）——限定在数字调度层

## 3. 工作流（Execution Workflow）

### 3.1 触发方式

- **对话式指令**："小明，这批德国仓的货为什么还没到""帮我看下哪些 SKU 快断货"
- **定时触发**：每小时扫一次全量在途轨迹；每天 8:00 跑一次断货预测；每周一汇总承运商表现
- **事件驱动**：检测到 ETA 延迟>24h、海关查验、FBA 拒收、可售天数<7 天 等任一信号即自动触发
- **上游 Agent 调用**：内部分析小明发现某 SKU 销量突增唤起断货检查；营销小明放量前唤起库存承载评估；合规小明清关补件时反向调用

### 3.2 工作流（主场景：供应链危机处置 — 对应增补模块场景剧本 B，3 步精简版）

> **Step 1：异常发现 & 影响评估**（合并原 Step 1+2，覆盖 LOG-Skill-01、02、03、04、06）
>
> ├── 1.1 实时监听与异常发现：聚合承运商 API / 货代 / 海外仓 WMS / FBA 入库 / ERP 库存 / 销售流水；识别 ETA 延迟 / 海关查验 / 拒收 / 破损 / 丢失 / 入库延迟 / 断货临界 7 类异常
>
> ├── 1.2 影响评估：ETA 偏差重新预测；受影响 SKU 可售天数推算；断货概率 + 预计断货日 + GMV/利润影响测算
>
> └── 异常路径：单一数据源失败 → 切备选源标"时效性"；销售/库存数据缺失 → 90 天均值兜底

> **Step 2：方案生成 & 跨 Agent 协同派单**（合并原 Step 3+4，覆盖 LOG-Skill-05、07、09、10）
>
> ├── 2.1 多方案生成：自动生成三套候选方案（继续海运 / 部分空运补发 / 全部切空运 / 海外仓调拨）；成本-时效-断货风险三维对比；联动内部分析小明重算 GMV/利润损失并给出推荐方案
>
> ├── 2.2 跨 Agent 协同派单：内部分析（受影响 SKU + 补货成本）/ 营销（可售库存 + 断货风险 → 广告守护模式）/ 合规（海关查验 + 补件清单）/ 经营评估（补发成本 + 资金缺口 → DE 周转资金任务）
>
> ├── ⚠️ **HITL-1：物流危机处置方案审批卡**（决策人=供应链负责人；附三方案 ROI 对比 + 推荐方案 + 跨 Agent 派单清单）
>
> └── 异常路径：路由报价库未覆盖 → 转人工询价并暂停 X 权限；下游 Agent 无响应 → 重试 3 次后转人工 ping

> **Step 3：执行 & 证据闭环**（合并原 Step 5+6，覆盖 LOG-Skill-07、08、09；含 X 权限）
>
> ├── 3.1 执行：内部任务流转 + Vault 写入 + 看板刷新（自动）
>
> ├── ⚠️ **HITL-2：执行确认卡**（决策人=供应链负责人；附费用/时效/影响范围；X 权限前置 — 货代下单 / 仓库调拨 / 索赔工单提交）
>
> ├── 3.2 证据闭环：自动组织索赔证据包（破损/丢失/拒收）→ 风险案例归档至 Compliance Vault → 承运商表现评分回写知识库
>
> ├── ⚠️ **HITL-3：索赔提交确认卡**（决策人=财务/法务；仅金额&gt;阈值时触发）
>
> └── 异常路径：货代/仓库系统驳回 → 自动定位驳回原因并回到 Step 2

> **3 步合计**：原 6 步压缩为 3 步，HITL 数量保持 3 个，Skill 调用数与覆盖能力**完全一致**——把"信号采集+影响测算"和"方案+派单"以及"执行+证据"三类紧密耦合的连贯动作合并到同一个主步骤的子动作中。危机处置闭环时长目标 ≤ 30 分钟由此达成。

### 3.3 HITL 节点汇总

| 节点 | 决策人 | 决策内容 | 决策依据卡片 |
|---|---|---|---|
| Step 2 处置方案审批 | 企业供应链负责人 | 选哪一套（海运/部分空运/全部空运/调拨） | 三方案 ROI 对比、断货预测、推荐方案理由、跨 Agent 派单清单 |
| Step 3 执行确认 | 企业供应链负责人 | 是/否调用货代下单/调拨/索赔工单 | 费用预算、生效路由、影响 SKU 清单、不可逆性提示 |
| Step 3 索赔提交 | 财务/法务负责人 | 是/否向承运商或保险公司提交索赔 | 证据包预览、金额、责任划分、过往案例胜算 |

### 3.4 下游交接

- ETA 延迟/断货预警 → **内部分析小明**（用于 GMV/利润/排名损失测算）
- 库存可售天数 + 补货 ETA → **营销小明**（用于广告节奏调整、爆款承接评估）
- 清关异常 + 单证缺失 → **合规小明**（双向：合规样板 §3.4 中"HS 归类结果 → 物流小明"形成闭环）
- 物流轨迹 + 采购单 + 运输真实性证据 → **经营评估小明**（用于 DE 报告供应链金融增信）

### 6.2 三层审计逻辑

- **第一层 · 数据采集**：LOG-Audit-01、06、08 — 每一次外部数据进出的原文留痕，配哈希
- **第二层 · 决策记录**：LOG-Audit-02、03、05 — AI 推理（ETA/方案/派单）的输入/输出/置信度全留痕，可回放
- **第三层 · 人工裁决**：LOG-Audit-04、07、09 — HITL 节点"谁、何时、看到了什么、点了哪个按钮"上区块链

### 6.3 KPI ↔ 证据映射

| KPI | 取数证据 |
|---|---|
| 异常发现提前量 | LOG-Audit-01 时间戳 vs 承运商状态变更时间 |
| 处置方案生成时长 | LOG-Audit-01 → LOG-Audit-03 时间戳差（Step 1 → Step 2） |
| ETA 预测准确率 | LOG-Audit-02 历史预测 vs LOG-Audit-06 实际到仓 |
| 索赔成功率 | LOG-Audit-08 索赔提交 vs 保险/承运商回执 |
| 物流成本节省率 | LOG-Audit-06 实际结算账单聚合 |
| 跨 Agent 协同闭环率 | LOG-Audit-05 派单/回传匹配率 |
| 步骤经济性达成度 | 主步骤数 ≤ 3 / HITL ≤ 3（架构层硬约束，每次任务复盘核对） |

### 6.4 与"幻觉"问题的关系

物流小明的所有对外输出（ETA 预测、方案推荐、索赔金额、断货日）必须能在 LOG-Audit-02、03 里找到推理来源——包括引用了哪些轨迹数据点、哪个历史样本、哪条承运商规则。**任何无法回溯到第一手证据的 AI 输出，工作流引擎自动拦截、不进入 Step 3 的 X 权限调用**。这是物流小明的硬约束。

### 7.2 交付物组合逻辑

- **看板（给老板看）**：在途态势看板 + 断货风险预测 → 老板 5 分钟掌握全局供应链态势
- **证据包（给保险/银行/审计看）**：索赔证据包 + FBA 入库复盘报告 + 资金缺口测算表 → 一键应对承运商扯皮、保险拒赔、银行尽调
- **数据接口（给下游 Agent 用）**：ETA / 库存 / 风险评分 API → 营销、内部分析、合规、经营评估实时调用

### 7.3 演示样例

> **演示场景**：增补模块场景剧本 B —— "海运 ETA 延迟 7 天 + FBA 即将断货 + 资金缺口"

> **输入**：物流小明自动检测到承运商 API 返回的 ETA 延迟告警（无需用户输入）
>
> **过程**：3 个 HITL 决策卡，全程约 18 分钟（现场 Demo 可压缩为 5 分钟）
>
> **输出**：
> - 1 份《物流异常处置卡》（含三方案 ROI 对比，推荐"30% 空运 + 70% 海运"混合方案）
